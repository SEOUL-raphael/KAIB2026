기술 배경: 이 페이지를 RAG로 보면

Retrieval · Augmentation · Generation
1. Retrieval

공개 페이지에서는 Supabase의 assembly_minutes_raw에서 날짜 범위와 필터 조건으로 회의록을 조회합니다. 페이지 자체는 브라우저나 LAN 서버에서 임베딩을 만들지 않고, 업로드된 데이터와 사전 적재된 결과만 사용합니다.

2. Augmentation

조회된 발언 묶음에서 위원회·발언자·안건·대표 발언을 함께 정리해 MiniMax에 전달합니다. 즉 모델은 인터넷 전체가 아니라 현재 공개 테이블에 올라온 회의록 범위에 근거해 답변합니다.

3. Generation

MiniMax는 검색된 컨텍스트를 바탕으로 브리핑 답변과 공개형 판단 로그를 생성합니다. 내부 추론 전체 대신, 사람이 검토 가능한 근거·경로·대표 발언 묶음을 함께 표시합니다.

동기화 경로

원천 SQLite 회의록은 Supabase 공개 테이블로 증분 동기화되고, 임베딩 작업은 별도 로컬 PC에서 수행한 뒤 결과만 업로드됩니다. 이 공개 페이지는 최신 meeting_date를 읽어 기본 조회 범위를 자동으로 맞춥니다.

왜 이 구조가 유용한가

검색과 생성이 분리돼 있어서, 사용자는 원문 목록으로 사실을 확인하고 MiniMax 브리핑으로 핵심을 빠르게 파악할 수 있습니다. 즉 “원문 근거 + 업로드된 데이터 + 모델 설명”이 같이 움직입니다.

현재 한계

공개 페이지는 실시간 임베딩을 만들지 않기 때문에, 벡터 검색 품질은 로컬 PC에서 미리 생성해 업로드한 임베딩 범위와 품질에 좌우됩니다. 즉 공개판은 사전 적재 결과를 소비하는 구조입니다.
