기술 배경: 이 페이지를 RAG로 보면

Retrieval · Augmentation · Generation
1. Retrieval

기본적으로는 Supabase의 assembly_minutes_raw에서 회의록 범위를 조회합니다. 같은 PC에서 로컬 Nemotron 임베딩 서버가 열려 있으면 질문을 1024차원 벡터로 바꿔 pgvector 유사도 검색을 먼저 수행하고, 실패하면 SQL 범위 조회로 내려갑니다.

2. Augmentation

벡터 검색 또는 날짜 범위 검색으로 찾은 발언 중 상위 근거 묶음을 구성하고, 위원회·발언자·안건 정보를 함께 MiniMax에 전달합니다. 즉 모델은 인터넷 전체가 아니라 현재 회의록 근거 묶음을 기반으로 답변합니다.

3. Generation

MiniMax는 검색된 컨텍스트를 바탕으로 브리핑 답변과 공개형 판단 로그를 생성합니다. 내부 추론 전체 대신, 사람이 검토 가능한 근거·경로·대표 발언 묶음을 함께 표시합니다.

동기화 경로

원천 SQLite 회의록은 Supabase 공개 테이블로 증분 동기화되고, Nemotron 1024 문서 벡터는 별도 pgvector 테이블로 적재됩니다. 이 공개 페이지는 최신 meeting_date를 읽어 기본 조회 범위를 자동으로 맞춥니다.

왜 이 구조가 유용한가

검색과 생성이 분리돼 있어서, 사용자는 원문 목록으로 사실을 확인하고 MiniMax 브리핑으로 핵심을 빠르게 파악할 수 있습니다. 즉 “원문 근거 + 벡터 검색 + 모델 설명”이 같이 움직입니다.

현재 한계

온라인 GitHub Pages는 외부 정적 페이지이기 때문에, 로컬 Nemotron 서버는 같은 PC에서 접속할 때만 직접 사용할 수 있습니다. 다른 기기에서는 자동으로 SQL 범위 조회 fallback이 동작합니다.
